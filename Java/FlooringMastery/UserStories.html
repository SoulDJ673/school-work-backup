User Stories

Story 1

The user will boot the program and be greeted with a menu. They will then promptly select the "Quit" option, and the program will prompt for them to save. They will respond with yes/no and the program will act according to their answer, then exit.

Story 2

The user will boot the program and be greeted with a menu. The user will select the option to create an order. The program will then display prompts asking for information such as the customer name, state of delivery, etc. An Order object will be created after all of this, which will calculate the total price automatically when constructed. The program will display the cost of the order before asking the user for confirmation. Declining the confirmation prompt will discard all data that was created relating to that order, while confirming the order will give it the latest available ID and save it to memory with that ID. The user will then be returned to the main menu, where they will select the option to quit, and be stopped by a prompt asking them if they'd like to save. Depending on whether or not they're in production mode, selecting "Yes" will save the order that is in memory to it's corresponding Order_XXYYZZ file. Being in training mode will throw an exception, but will still allow the user to quit without saving.

Story 3a

The user will boot the program and be greeted with a menu. The user will select the option to display orders for a date. They will enter the date that the previous order (from Story 1) was set for. The user will then select the option to remove an order. The program will first prompt the user to enter a date, then an order ID. The user will respond with the date of the aforementioned order, and the ID of the order that was returned. The program will check if there is an order that matches the information provided. If there is, the user will be prompted for confirmation. If the user confirms, a call to the orderDao will remove the order from memory. Otherwise, the order will remain in memory. Either way, the user will be returned to the main menu. If there isn't an order that matches the provided information, the user will be shown an error message before being returned to the main menu. The user will select the option to quit, being shown a prompt to save before exiting. Our user is in production mode, so when they confirm the save the order will be removed from the order file.

Story 3b

The user will boot the program and be greeted with a menu. The user will select the option to display orders for a date. They will enter the date that the previous order (from Story 1) was set for. The user will then select the option to edit an order. The program will first prompt the user to enter a date, then an order ID. The user will respond with the date of the aforementioned order, and the id of the order that was returned by the display command. The user will then go through various prompts, be shown the previous information about the Order at the corresponding prompt, skipping all of them until they arrive at the area field. Here they will change the value of the area of material they ordered, then continue on through the prompts until they reach the end. The program will call the recalculate() method in the Order object, display the new price, and ask the user to confirm the order. The program will save the Order memory if the user confirms, and it won't if the user doesn't consent. The user will be returned to the main menu. The user will select the option to quit, be shown a Save confirmation prompt, and select yes. If they hadn't, their newly edited order will not be saved to it's file. However, since they confirmed the save, the data in the file has been replaced with the new data they have given.


Sprint 1 (11/30)

  • The user will be able to successfully boot the program.
  • The user will be able to display all orders for a specified date.

    • Sprint 2 (12/06)

  • The user will be able to add an order to memory.
  • The user will be able to remove an order from memory.
  • The user will be able to edit an order in memory.
  • The user will be able to toggle production/training mode.

    • Sprint 3 (12/12)

  • The user will be able to save orders to a file named by date without replacing an existing file (if it exists).
